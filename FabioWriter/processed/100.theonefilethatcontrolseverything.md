# The One File That Controls Every Claude Code Session (And How to Write It Correctly)


Most developers either ignore CLAUDE.md entirely or populate it with unrelated content. The key point is that there’s a specific Claude code file loaded first in a session—before your initial prompt, any code, or even reviewing your codebase.

The file named`CLAUDE.md` is located at the root of your project. Claude considers it the definitive source for everything that comes after. If you get it right, you can brief a senior engineer about your project in just 30 seconds. If you get it wrong or omit it, you risk spending an hour with Claude trying to figure out your build commands, refactoring files you didn't specify, and committing a confusing, large commit that spans 47 files.

## Why Your Current CLAUDE.md Probably Isn’t Working

There are three failure modes. Almost every bad `CLAUDE.md` hits at least two of them.

**Failure mode 1: Too long.**

> Claude Code’s system prompt already uses about 50 instructions from the model’s capacity. Most models can reliably handle 150–200 instructions at once. This suggests your model has a remaining budget of roughly 100–150 instructions before Claude begins dropping some — not because it's ignoring you, but because it can't reliably maintain that much context.

If your file is 200+ lines, Claude isn’t being difficult. You’ve exceeded the budget.

**Failure mode 2: Wrong content.**

> The most frequent error is including instructions that Claude can deduce independently. Personality directives such as “act as a senior engineer” or “think step by step” fail to significantly alter its behavior. Vague advice that applies universally to all projects only introduces confusion without providing clear guidance. Each instruction that does not help prevent a particular mistake is essentially unnecessary clutter.

**Failure mode 3: No hierarchy.**

Many people are unaware that there are three distinct locations to place instructions, each with its own scope.

~/.claude/CLAUDE.md          # Global — applies to every project  
.claude/CLAUDE.md            # Project — shared with team, committed to git  
./CLAUDE.local.md            # Local — personal overrides, gitignored

Putting everything into a single file makes all three sections unnecessarily long. The 'Global' section is for rules you’d apply across all projects (like "always use try/catch for async calls"), the 'Project' section contains stack-specific context your team requires, and the 'Local' section is for your personal preferences. Using all three helps keep each file concise and focused.

## The 5 Sections That Actually Do Something

After going through dozens of production `CLAUDE.md` files from open-source projects, Anthropic's own docs, and community best practices, every effective file covers exactly these five things.

## Section 1: Critical Commands

Claude doesn’t know how to build, test, or lint your specific project. Tell it.

\## Commands  
\- Dev: \`npm run dev\`  
\- Build: \`npm run build\`  
\- Test single file: \`npm test -- path/to/file\`  
\- Test all: \`npm test\`  
\- Lint + fix: \`npm run lint:fix\`  
\- Type check: \`npx tsc --noEmit\`

Short and specific. Without this section, Claude will attempt `npm test` on a project that uses `pnpm vitest` and spend three turns debugging a command that was never going to work. This section alone saves significant time on every session.

## Section 2: Architecture Map

Claude starts every session with zero knowledge of your codebase. Give it a map — not a full directory tree, just enough to know where things live and what belongs where.

\## Architecture  
\- src/lib/services/     → all business logic  
\- src/components/       → stateless UI components only  
\- src/lib/store/        → global state (Zustand)  
\- src/app/api/          → API routes, no business logic here  
\- Database access only through Server Actions or API routes

This prevents Claude from putting business logic in a UI component, dropping state management in the wrong folder, or creating an API route that does things API routes in your project shouldn’t do. One section, five lines, eliminates an entire class of architectural mistakes.

## Section 3: Hard Rules

This is the most important section in the file.

Every rule here should pass a single test: _Would removing this line cause Claude to make a mistake?_ If the answer is no, the line doesn’t belong here.

\## Rules  
\- NEVER commit .env files or secrets  
\- All async calls must use try/catch  
\- Use functional components only, no class components  
\- Prefix commits: feat:, fix:, docs:, refactor:  
\- All PRs must pass \`npm run verify\` before merge  
\- Static export only, no SSR (deployed to S3)  
\- IMPORTANT: run type check after every code change

Two things worth noticing here.

First, negative rules are just as important as positive ones. “Never commit .env files” prevents a potentially catastrophic mistake, while “Use functional components” avoids a stylistic issue. Both are essential.

Second, emphasis markers actually work. Anthropic’s own documentation confirms that prefixing a rule with `IMPORTANT` or `YOU MUST` meaningfully improves how reliably Claude follows it. Use `IMPORTANT` for the one rule Claude keeps breaking on your project.

Keep this section under 15 rules. More than that and the important ones get diluted.

## Section 4: Workflow Preferences

This section answers the question: _how do you want Claude to behave while it works?_

\## Workflow  
\- Ask clarifying questions before starting complex tasks  
\- Make minimal changes, don't refactor unrelated code  
\- Run tests after every change, fix failures before moving on  
\- Create separate commits per logical change, not one giant commit  
\- When unsure between two approaches, explain both and let me choose

The second line — “make minimal changes, don’t refactor unrelated code” — is often what people wish they had included after Claude rewrote an entire file just to modify one function. The fifth line stops Claude from making architectural choices on its own. These are workflow agreements, not personal characteristics. They influence behavior in ways you’ll see right away.

## Section 5: What NOT to Include

Equally important is what you leave out. These things actively hurt your `CLAUDE.md` by consuming instruction budget without contributing value:

\## Don't include:  
\- Personality instructions ("be a senior engineer")  
\- Code formatting rules your linter already handles  
\- @-imports that embed entire docs into every session  
\- Duplicate rules (if global says "run tests," project doesn't repeat it)  
\- Anything Claude will learn on its own via auto memory

That last point is underrated. Claude maintains its own notes at `~/.claude/projects/<project>/memory/`. Run `/memory` in any Claude Code session to see what it has already learned about your project from previous sessions. Don't waste `CLAUDE.md` lines documenting things Claude figured out after one hour of work.

## The Full Template (Copy This)

Here’s a production-ready `CLAUDE.md` you can copy and adapt. Under 60 lines. Covers everything Claude needs, nothing it doesn't.

\# CLAUDE.md  
  
\## Project  
\[One line: what this project does and who uses it\]  
\## Stack  
\[Framework, language, database, deployment target\]  
\## Commands  
\- Dev: \`\[command\]\`  
\- Build: \`\[command\]\`  
\- Test single: \`\[command\] -- \[path\]\`  
\- Test all: \`\[command\]\`  
\- Lint: \`\[command\]\`  
\- Type check: \`\[command\]\`  
\## Architecture  
\- \[folder\] → \[what lives here\]  
\- \[folder\] → \[what lives here\]  
\- \[folder\] → \[what lives here\]  
\- \[file\]   → \[what this file does\]  
\## Rules  
\- \[Rule that prevents a specific mistake\]  
\- \[Rule that prevents a specific mistake\]  
\- \[Rule that prevents a specific mistake\]  
\- IMPORTANT: \[The one rule Claude keeps breaking\]  
\## Workflow  
\- \[How you want Claude to approach tasks\]  
\- \[Commit conventions\]  
\- \[Testing expectations\]  
\- \[When to ask vs when to act\]  
\## Out of scope  
\- \[Things Claude should not touch\]  
\- \[Files that are manually maintained\]  
\- \[Integrations Claude shouldn't modify\]

Delete sections that don’t apply to your project.

## The Lines With the Highest Impact

After testing various configurations, these particular lines led to the greatest enhancement in output quality — each addressing a common, predictable error.

\# Lines with the highest impact (and why):  
  
\- IMPORTANT: run type check after every code change  
  → prevents Claude from shipping broken types silently  
\- Make minimal changes, don't refactor unrelated code  
  → prevents Claude from rewriting your entire file for a one-line fix  
\- Create separate commits per logical change  
  → prevents the 47\-file monster commit  
\- When unsure, explain both approaches and let me choose  
  → prevents Claude from making architectural decisions for you  
\- Static export only, no SSR  
  → prevents Claude from adding server-side code to a static site

Observe the pattern: each line is associated with the mistake it avoids. That’s the test. If you cannot identify the mistake a rule prevents, then that rule likely doesn’t belong in the file.

## The Mindset Shift That Makes This Click

People treat `CLAUDE.md` like a wish list. They write what they want Claude to be — experienced, thoughtful, careful, rather than what they need it to know.

A wish list doesn’t change behaviour. A technical brief does.

Focus on stack, commands, architecture, rules, and workflow. Everything else is noise vying for attention with the essential instructions. Keep it within 80 lines. Review it if Claude makes an error. Add a rule each time you identify a mistake you’ve already fixed twice.

The file compounds over time. A good `CLAUDE.md`in month one means you stop repeating yourself by month two. By month three, Claude is running your project workflows the way you'd run them yourself — because you took 30 minutes once to write down exactly how that looks. That 30 minutes is the highest-leverage thing you can do before opening Claude Code.
