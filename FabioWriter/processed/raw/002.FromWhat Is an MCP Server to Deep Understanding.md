## Introduction

When I first heard the term “MCP server” — I didn’t get it. At all. No matter how many fancy definitions or technical explanations I read, it just didn’t click. It felt like one more tech acronym tossed into a pile I was supposed to pretend to understand.

But instead of faking it, I started asking questions. Not sophisticated ones. Just raw, basic, brutally honest ones.

And what happened over the next 48 hours? A transformation.

This is the unfiltered story of how I learned what an MCP server is, how it works, why it matters, how the architecture is built, how the roles are distributed between AI, client, and server — and how _you_ can understand it too.

It’s not just a technical guide. It’s a mindset. A method. A story of asking, doubting, connecting, cursing, confronting contradictions — and finally… understanding.

![](https://miro.medium.com/v2/resize:fit:700/1*bIi7Z-JXZUz-orrD1yyMcQ.png)

## Step 1: What Is an MCP Server?

My first question was blunt:

> _“What is MCP server? I can’t understand it.”_

The answer I got:

> An MCP server is a bridge between AI models and external tools or data sources. It lets the AI send requests in a standard way, and it takes care of translating those into the specific tool’s logic (like APIs, file systems, etc).

I could follow that. Bridge. Middleman. “Universal adapter” came up.

But then came the real questions. Because _bridge to what?_ and _from what exactly?_

## Step 2: Doesn’t the Developer Still Have to Code?

> _“You say this saves work for developers — but don’t we still need to write code to connect to APIs and tools?”_

Answer: Yes. But here’s the key:

-   Without MCP: Every developer writes custom code _inside the AI system_ for every tool it needs.
-   With MCP: You write that connection code _once_, into an MCP server — and now _any_ AI client that speaks MCP can reuse it.

That’s the benefit:

-   One server per tool
-   Reused by many AIs

## Step 3: Isn’t That What REST APIs Do?

> _“Why not just let the AI call GitHub’s REST API directly?”_

Now THIS one bothered me.

I’ve worked with APIs. Why not just hit them directly?

Then Grok explained:

-   REST APIs are all over the place — OAuth here, tokens there, XML, JSON, various formats.
-   An AI has to be taught each API — the headers, the params, error codes.

But with MCP:

-   The AI sends ONE standard format
-   The MCP server knows the REST API and does the messy translation

So the AI says:

{  
  "tool": "github",  
  "action": "create\_issue",  
  "params": {  
    "repo": "my/project",  
    "title": "Bug"  
  }  
}

…and the MCP server figures out the endpoint, handles tokens, formats the call.

That’s when I finally got it:  
**MCP is like a universal remote control.** The AI just points and says what it wants.

## Step 4: What Exactly Talks to What? Where Does the Client Lie?

At this point, the pieces felt scattered.

I asked:

> _“Where does the MCP client lie? Who runs it? Is Claude desktop the AI + client ? Is it inside the AI?”_

Here’s the final answer I arrived at (after a dozen clarifying replies):

-   The **AI model** (like Claude or your own LLM) generates the intent: _“create issue on GitHub”_
-   The **MCP Client** is the module that takes that structured request and sends it across to the server. It may be part of the AI app (like Claude Desktop) or live separately.
-   The **MCP Server** is where the actual tool-specific logic resides. It connects to GitHub, Gmaps, Filesystem, etc.

Sometimes, an AI _acts as both the AI and the client_ — like Claude does in a hosted environment.

Other times, you run a standalone AI and separately install an MCP client that talks to your local or remote servers.

That separation — AI ↔ client ↔ server — finally helped me understand the architecture.

## Step 5: Isn’t This Still Just One Request per Tool?

I hit another wall:

> _“So I still need to make separate requests to different MCP servers — isn’t this just as fragmented?”_

Answer: _Yes_, each tool has its own MCP server. But the _request format is the same._ That’s the win.

Instead of:

-   learning GitHub’s API
-   then Google Maps’
-   then Slack’s

You send the same JSON format to all:

{  
  "tool": "gmaps",  
  "action": "get\_directions",  
  "params": {  
    "origin": "Kochi",  
    "destination": "Bangalore"  
  }  
}

Same structure. Different tool. Reusable code.

And most importantly — the AI never learns the APIs. The servers abstract it all.

## Step 6: Prompt vs API Call?

Another one I couldn’t let go:

> _“Is this prompt-based or structured API request?”_

Turns out — it’s both.

In many setups:

-   The AI creates the _intent_ as a natural language string (“create GitHub issue with title ‘Bug’”).
-   The MCP client (or an interpreter layer) turns that into JSON
-   That JSON is sent as a proper MCP request

So you get the simplicity of prompts _and_ the structure of APIs.

It’s why this works well for LLMs. They speak prompts. The client translates.

## Step 7: Can AI Ask What Tools Are Available?

This blew my mind. I asked:

> _“How does the AI know what servers/tools exist? How does it know what parameters to pass?”_

And MCP supports _introspection_.

The AI (or client) can ask:

{  
  "action": "list\_tools"  
}

The MCP server replies:

\[  
  {  
    "tool": "github",  
    "actions": \[  
      {  
        "name": "create\_issue",  
        "params": { "repo": "string", "title": "string" }  
      }  
    \]  
  }  
\]

So yes — the AI can dynamically learn:

-   What tools are present
-   What actions are available
-   What data each action expects

That was a killer feature for me.

## Step 8: The Final Flow — From AI to Real World

Here’s what finally made the whole thing gel:

\[User\]  
   ↓  
\[AI (Claude)\] — decides intent  
   ↓  
\[MCP Client\] — formats structured MCP request  
   ↓  
\[MCP Server (e.g. GitHub)\] — talks to API, performs action  
   ↓  
\[Response\] — comes back up to AI  
   ↓  
\[User sees result\]

It’s prompt-to-action — with an invisible backend doing the API juggling.

## Final Thoughts: Learning This Wasn’t Smooth — It Was Human

I didn’t just get this by reading documentation. I ranted. I doubted. I cursed. I re-asked questions that were “already answered.”

I asked:

-   “Where does the client lie?”
-   “Is Claude both the client and the AI?”
-   “Why not just use REST?”
-   “So I still make separate calls per tool? What’s the point?”
-   “How does the AI know what to pass?”

Each time, I wasn’t just being pedantic. I was being _thoroughly human_. That’s what learning really looks like.

And now?

I can say I _genuinely understand_ what MCP is.

And you can too.

Not because you memorized terms. But because you followed a line of reasoning — until every contradiction collapsed.

That is the best kind of learning.

## Final Thoughts

-   **MCP** connects AIs to real-world tools.
-   It separates AI from tool-specific logic.
-   You build one server per tool — reusable by any AI.
-   The AI talks to all tools via **one standard format**.
-   The **MCP client** sends the AI’s structured request to the right server.
-   Servers can be discovered and introspected.
-   It’s not just a REST replacement — it’s a system of separation, clarity, and reusability.

And it all starts with asking the next question.  
Even if it’s “Where does the client lie?”
