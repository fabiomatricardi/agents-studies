# Claude Code hooks explained: the missing layer between prompts and production

## Why memory and rules are not enough and how hooks enforce behaviour at the right moment.


This week, I found myself trying to make behaviour consistent across a few real workflows when working with Claude Code. I thought Claude _knew_ what it was supposed to do (the rules were there, I had a clear set of instructions in the claude.md and agents readily set up). And yet, sometimes it followed them perfectly, and other times, I had to ask Claude to run a specific follow up task.

That’s when a colleague pointed me to hooks.

The problem was not memory. It was not prompting either. It was the fact that I was relying on a probabilistic system to behave deterministically.

This is exactly the gap hooks are designed to fill.

In this post, I will break down what Claude Code hooks actually are, how they work, and — more importantly — how to think about them. Not as another feature, but as the missing layer between prompts and production systems, where behaviour stops being “best effort” and starts becoming enforced.

If you are trying to make LLM-based workflows reliable in real projects, this post is for you.

## What this blog will cover

-   **Why hooks exist at all.** Why Claude knowing the rules is not enough, and where things break in practice when behaviour is left to the model’s judgement.
-   **What hooks are, in one clean mental model.** Hooks as lifecycle control points: not memory, not guidance, but explicit moments where the system can enforce, validate, or react.
-   **The hook events that matter most in practice.** The small subset of events that actually drive most real workflows, and how to think about them without memorising the full lifecycle table.
-   **How hooks actually work technically.** From event → matcher → structured input → handler.
-   **Advanced hooks (when you want Claude to double-check itself).** A few patterns where hooks go beyond automation and start enforcing reasoning, using prompt or agent-based checks.
-   **Real examples useful for in data science and ML repos.** From guardrails and notifications to deferred validation and reproducibility checks, with concrete patterns that translate directly to ML and data workflows.

Let’s get started!

## Why hooks exist at all.

In previous posts, we saw that Claude can [remember things (](/data-science-collective/you-probably-dont-understand-claude-code-memory-3cb26955a8ea)[_memory_](/data-science-collective/you-probably-dont-understand-claude-code-memory-3cb26955a8ea)[)](/data-science-collective/you-probably-dont-understand-claude-code-memory-3cb26955a8ea) and follow [structured guidance (](/data-science-collective/how-claude-code-rules-actually-work-8b01faaa4d9d)[_rules_](/data-science-collective/how-claude-code-rules-actually-work-8b01faaa4d9d)[)](/data-science-collective/how-claude-code-rules-actually-work-8b01faaa4d9d). That already gets you quite far. You define how you want it to behave, and in many cases, it does exactly that.

But, if you have [followed my LLM series](/data-science-collective/llms-explained-part-1-the-3-layer-framework-behind-chatgpt-friends-6189c3fa8ae3), you know that LLMs are probabilistic, and Claude is no different. It decides what to do next based on context, patterns, and trade-offs. Even when it “knows” a rule, applying it at the right moment is still a decision, not a guarantee.

That distinction matters more than it first appears. Basically because there is _BIG_ a difference between:

-   a system that _usually behaves correctly_
-   and a system where certain things _always happen_

Anthropic’s own documentation makes this distinction quite explicit: [**hooks are designed to introduce**](https://code.claude.com/docs/en/hooks?utm_source=chatgpt.com) [**_deterministic_**](https://code.claude.com/docs/en/hooks?utm_source=chatgpt.com) [**control at specific points in the lifecycle**](https://code.claude.com/docs/en/hooks), rather than relying on the model to remember or prioritise the right behaviour

This is the gap hooks fill. Hooks exist because some behaviours should not depend on judgement at all.

## What hooks are, in one clean mental model.

We briefly introduced that hooks bring **deterministic control at specific points in the lifecycle**. The key question is: _what does that actually mean in practice?_

Think of Claude as operating in a sequence of steps (a.k.a. a lifecycle). A prompt is submitted, tools are called, files are edited, a response is produced, a task is marked as complete.

Each of these moments is an **event**. And hooks let you attach logic to those events.

### A simple example

Imagine you want to enforce the rule:

> _“After code changes, the README must be updated.”_

You could write this in `CLAUDE.md`.

But, there is a problem: you are relying on the model to **remember and prioritise** that rule. Sometimes it will. Sometimes it will not — especially as context grows, tasks get complex, or attention shifts.

Ideally, what we would like is that when the event “code changes are done” is complete, then an action is triggered. In other words, we want a guaranteed behaviour that the LLM will, not only make the changes, but also check the documentation and update it appropriately.

## The hook events that matter most in practice.

At this point, we have been talking about “lifecycle events” as the moments where hooks introduce deterministic behaviour. But, of course, the important question is: **_What events can we actually use?_**

Anthropic [provides a full lifecycle diagram and event table](https://code.claude.com/docs/en/hooks?utm_source=chatgpt.com), and it is fairly comprehensive. As of time of writing, there are 22 events that can be triggered. You definitely _don’t_ need to memorise all of them. I personally find it much easier to reason about them by grouping them into 5 categories:

Each group represents a different moment in the lifecycle:

-   when a session starts or context changes
-   before an action happens
-   while something is running
-   after something happens
-   when work is about to be considered “done”

Once you think in these terms, the question shifts from:

> “Which event should I use?”

to:

> “At what point in the workflow do I want to intervene?”

### The five events worth knowing first

If you are just getting started, you can safely focus on a small subset of events (highlighted with ⭐ in the table):

-   **SessionStart** — initialise context and setup
-   **PreToolUse** — intercept and control actions before they happen
-   **PostToolUse** — react immediately after an action
-   **Stop** — enforce checks before the response completes
-   **TaskCompleted** — enforce a stricter definition of done

These 5 cover most real-world use cases: guardrails, validation, quality control, and context management. Everything else tends to be either supporting behaviour (notifications, logging) or more advanced workflows (multi-agent coordination, MCP interactions).

### Back to our README example

Let’s revisit the earlier rule: _“After code changes, the README must be updated.”_

There is no single “correct” event for this. It depends on _when_ you want to enforce it.

-   Using **PostToolUse** → Check immediately after file edits happen
-   Using **Stop** → Perform a final validation before returning the result
-   Using **TaskCompleted** → Enforce it as part of the definition of done

Each choice leads to slightly different behaviour, even though the rule itself is the same. That is the key idea:

> _Hooks are not just about_ what _you enforce, but_ when _you enforce it._

## How hooks actually work technically.

At a technical level, hooks are quite simple.

A hook starts with an **event**, can optionally be narrowed with a **matcher**, receives **structured input** from Claude Code, and then executes one or more **handlers**. Something like this sequence:

event → matcher → structured input → handler → outcome

That sounds abstract, so let’s make it concrete.

### The basic JSON structure

At its simplest, a hook configuration looks like this:

{  
  "hooks": {  
    "PostToolUse": \[. \# <--- This is the lifecycle event  
      {  
        "matcher": "Edit|Write",  \# <--- This is the matcher (regex style or similar)  
        "hooks": \[  
          {  
            "type": "command",    \# <--- type says what kind of handler will run  
            "command": "/path/to/check-readme-after-edit.sh"  \# <-- the actual action being executed  
          }  
        \]  
      }  
    \]  
  }  
}

In this example, the hook listens for the `PostToolUse` event, filters it so it only applies to the `Edit` or `Write` tools, and then runs a shell command when that condition is met.

We have already covered the lifecycle event above, so let’s look at the matcher, the structured input and the handler in a bit more detail.

### The matcher

A matcher is how you avoid running the hook all the time (or in another words, how you specify _what determines_ that a hook should run).

Anthropic documents matchers as regex filters, and the exact field they filter depends on the event type. For tool-related events such as `PreToolUse`, `PostToolUse`, and `PostToolUseFailure`, the matcher is applied against `tool_name`. Some events such as `Stop` and `TaskCompleted` do not support matchers at all and always fire.

That means:

-   `Edit|Write` matches either tool
-   `Bash` matches only Bash
-   `mcp__.*` could match MCP tools
-   no matcher means “run every time” for matcherless events

This is why matchers matter so much in practice. They are what make a hook precise instead of noisy.

### The structured input

So, the **when** (ie, `PostToolUse`) and **what** (ie, `Edit|Match`) tell Claude that it should do something. But before doing something, Claude (or the hook) needs to know what just happened.

When Claude Code triggers a hook, it does not just say: _“Hey, something happened”._ It sends your script a **structured summary of the event**. You can think of it like a receipt:

-   what just happened
-   which tool was used
-   what inputs were passed
-   what files were involved

And this “receipt” is written in **JSON**. It we look back at our hook example, when the `PostToolUse` event fires and the matcher (`Edit|Write`) matches, then Claude sends something like:

{  
  "event": "PostToolUse",  
  "tool\_name": "Edit",  
  "tool\_input": {  
    "file\_path": "src/main.py"  
  }  
}

This is the **structured input**.

This structured input helps Claude point to the specific changes what where made, so that your script does not have to scan files randomnly or rely on the model memory or guess what happened. IIt is reacting to a **precise, structured description of the event**.

### The handler

Finally, we get to what actually runs once the event and matcher line up. Anthropic currently supports 4 hook types:

-   **command** → run a local script (most common, easiest to start with)
-   **http** → send the event to an external service (APIs, logging, Slack, etc.)
-   **prompt** → run an LLM-based check or transformation
-   **agent** → delegate the task to another agent

In practice, most early hooks you will see are `command` hooks because they are easy to understand and debug. A small shell script or Python script is usually enough for the first wave of useful automations.

### Back to our README example

We already showed an example above when the `type = command` and the idea was to run a shell script. Here is another example, where we could use `agent` or `prompt` to do something,

{  
  "hooks": {  
    "Stop": \[  
      {  
        "hooks": \[  
          {  
            "type": "agent",  
            "prompt": "Check if README.md reflects recent code changes. If not, update it accordingly."  
          }  
        \]  
      }  
    \]  
  }  
}

### Where does this JSON live?

Hooks use the same JSON settings format as Claude Code settings more broadly, and Anthropic documents them as part of `settings.json`. They can be configured at user level, project level, and in managed settings; plugins can also ship hook-like configuration using the same structure where supported

### Wait, but can’t I simply ask Claude to write hooks for me?

Of course! And that is what I personally do.

But, knowing the structure is still important, because you need to be able to evaluate what Claude produced.

You want to be able to look at a generated hook and ask:

-   Is this attached to the right event?
-   Is the matcher too broad?
-   Is this observing, blocking, or modifying?
-   Does the handler actually inspect the right input?
-   Is this safe enough to run on my machine?

That is the real skill: not typing the JSON from memory, but understanding the control surface well enough to review it properly.

## Advanced hooks (when you want Claude to double-check itself).

So far, most of the examples you will see online are **command hooks**, small scripts that run at specific points in the lifecycle.

That is a great place to start. But it is also a very narrow view of what hooks can do.

Hooks are not limited to shell scripts or simple automation. They can also trigger **prompts and agents**, which means they can introduce an extra layer of reasoning into your workflow (we briefly saw that in the readme example above).

At that point, hooks stop being just _automation_ and start becoming **checkpoints for thinking**.

### From automation to validation

There is a useful distinction here.

Some checks are mechanical:

-   “Did `README.md` change?”
-   “Did tests run?”
-   “Was this file edited?”

These are well handled by scripts.

But many real-world checks are not mechanical. They are **semantic**:

-   “Is the README actually up to date with the changes?”
-   “Does this implementation match the original request?”
-   “Is anything important missing before we consider this done?”

This is where prompt and agent hooks become useful.

They allow you to say:

> _“Pause here, and think again before moving forward.”_

### What changes when you use prompt or agent hooks

Instead of just running a script, you can:

-   run a **targeted prompt** to validate a specific condition
-   trigger an **agent** to review, improve, or complete work
-   call external systems through MCPs (Slack, Confluence, APIs, etc.)

This turns hooks into something closer to a **control layer for workflows**, not just a trigger system.

And because hooks are tied to lifecycle events, those checks happen **consistently, every time**.

## Real examples useful for in data science and ML repos.

Data science workflows can inherit a lot of the ideas that work well for software engineering (at the end of the day, we also write code). But I also want to show you a couple of examples that might trigger some ideas for your modelling workflows.

### 1) Coding conventions (the typical use case)

One of the simplest — and most useful — patterns is to run coding conventions **when the work is done**. Using a `Stop` or `TaskCompleted` hook, you can run:

-   tests
-   linters
-   type checks

As I mentioned, the typical use case everyone talks about. Still, the one that, when I had to code a lot, I always forgot to do `make lint` or `make ruff` and it cost me so maybe re-commits to git… These hooks have made my life so much easier.

### 2) Automatically updating experiment documentation

Documentation is one of those things that everyone agrees is important… and almost always gets skipped. Hooks give you a way to make it part of the workflow.

For example, after a training run, you can trigger a hook that:

-   extracts key parameters
-   captures results (metrics, outputs)
-   updates a README or experiment log

This can be done with a simple command hook, or with an agent that formats the update more cleanly. In fact, if you use Databricks (via MCP or CLI), you can read the stored artifacts and ensure that all key metrics have been logged, tags have been applied, etc.

The key idea is to ensure the quality of the documentation stops being a manual task and becomes a side-effect of doing the work.

### 3) ML training run sanity check

I personally really like this one.

Instead of just checking that something ran, you can use a **prompt or agent hook** to ask:

-   Are these metrics reasonable?
-   Is there evidence of overfitting?
-   Does the evaluation match the original objective?

For example, using a `Stop` hook:

{  
  “hooks”: {  
    “Stop”: \[  
      {  
        “hooks”: \[  
          {  
            “type”: “agent”,  
            “prompt”: “Review the latest model results. Check whether the evaluation is sound, whether there are signs of overfitting, and whether the conclusions are justified.”  
          }  
        \]  
      }  
    \]  
  }  
}

In practice, you would combine this with simple conventions:

-   only run this when training scripts were executed
-   or when metrics files were produced

The result is a lightweight “second opinion” built into your workflow.

Not perfect, but often enough to catch obvious issues before you move on.

### 4) Notifications for long-running workflows

Many data workflows are asynchronous: model training, data scraping, large transformations… and then can take a long time.wait.

Hooks can make this much easier to manage. For example, using a `Stop` or `Notification` hook, you can:

-   send a Slack message
-   trigger a webhook
-   log completion to an external system

This is particularly useful when combined with agents or MCP integrations.

Instead of checking manually: “Is it done yet?”, you let the system tell you.

## Closing thoughts

If there is a key idea for you to take away, it’s the following:

> _Hooks are_ **_not just_** _about telling the model_ **_what_** _to do. They are_ **_about_** _deciding_ **_when something must happen_**_._

When you start thinking in terms of lifecycle control rather than instructions, behaviour stops depending on whether the model “remembers” a rule and critical checks happen exactly when they should.

Hooks basically provide you the tool for **well-placed control points in the system**.

Once you internalise that, hooks stop feeling like a niche feature and start feeling like what they really are: **a control layer for building reliable AI systems.**
