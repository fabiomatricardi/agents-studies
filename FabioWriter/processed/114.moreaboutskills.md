Skills are specialized instructions you may want to provide to Claude in certain circumstances, but may not always be relevant. Slash commands are specialized commands you may want to invoke manually in certain circumstances. Sub-agents are specialized personas that you may want to offload specific types work to fresh contexts backed by a full agent instance.

So use subagents to keep your context clean when you want an actual full agent instance to perform something without poisoning your main context (code reviewer, skeptic, testing specialist, language/framework specialist, architect, etc). Use skills when you want to sometimes provide instructions to your primary agent with a relevant skillset. Use slash commands when there are things you specifically know you’ll want to invoke at certain points.

This is a good answer!

You can tell Claude to use a skill, but you can also ask Claude to "invoke all relevant skills" and Claude will then consider what skills to use. This eliminated the load of you having to tell Claude what to do. You can make this a SlashCommand.

SlashCommands can be invoked by the user but Claude can also invoke SlashCommands. For example you can create a git check in slash commands that standardizes how you check in. You can tell Claude to use this whenever they check in in your CLAUDE.md and the Claude will respect this.

Sun agents can be specialized personas but that is only half the value. The other half is context isolation (the sub agent doesn't bloat the main agent's context). This makes responses faster and cheaper particularly when working on a problem that requires a lot of expertise. Break it down into components and let agents work on it in parallel.

Bonus tips: * My commands are all 100% structured YML this facilitates implementing sequences and conditional actions reliably and efficiently. * Use Hooks to deny tool calls unless a requisite skill is equipped when agents ignore documented best practices. The pattern I use for example is toolname-allowed. So a bash-allowed skill gates bash access.

Claude is able to invoke multiple skills in a single message so a skill assessment slash commands, combined with tool-gating skills can efficiently get Claude juiced up with the knowlede it needs to be successful.

This is highly token efficient because Claude only loads the information it needs to be successful. This concept is called progressive disclosure.

A few months ago giant context windows and giant files with 1000s of best practices was the norm. The developing new norm is an assessment process that efficiently gives Claude just the context it needs to be successful.

Yeah tbh I find subagents the most useful/powerful. Skills are just an automated way of providing extra context to your agent. You could always paste/type instructions for skills, it’s just time consuming if you’re applying the same “skill set” frequently. Really for skills vs slash commands, the main difference effectively is that Claude will discover skills and determine when to use them on its own, where slash commands are manually prompted. That makes slash commands more suited for things like workflow automation. There’s certainly a lot of overlap though.
