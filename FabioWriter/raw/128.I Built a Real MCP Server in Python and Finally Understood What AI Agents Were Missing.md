## What Is MCP in Simple Words?

MCP means **Model Context Protocol**.

Let’s keep it simple.

**MCP is like a USB-C port for AI applications.**

Before USB-C, every device had a different cable.

After USB-C, many devices could connect through one common standard.

MCP tries to do something similar for AI agents.

Instead of every AI app building a different way to connect with tools, files, databases, and APIs, MCP gives a common structure.

With MCP, an AI agent can connect to an MCP server.

That MCP server can expose:

-   Tools
-   Resources
-   Prompts
-   Data
-   Business logic
-   External actions

So instead of putting everything inside the AI agent, we can separate the system into two parts:

AI Agent / MCP Client  
        ↓  
MCP Server  
        ↓  
Tools, Files, APIs, Database, Business Logic

This is powerful because the AI agent does not need to know every internal detail.

It only needs to know what tools are available, what inputs they need, and what output they return.

That is a big shift.

## MCP Tools, Resources, and Prompts

To understand MCP properly, you need to understand three basic ideas:

-   Tools
-   Resources
-   Prompts

These three concepts are very important.

## Tools

Tools are actions.

A tool does something.

Examples:

-   Calculate price
-   Read file
-   Search database
-   Create ticket
-   Send email
-   Check deployment status
-   Generate report
-   Fetch API response

In our example, these are tools:

get\_project\_status()  
estimate\_project\_cost()  
summarize\_project\_risk()

Tools allow the AI agent to act.

## Resources

Resources are context.

A resource gives information.

Examples:

-   Files
-   Documents
-   Database rows
-   API data
-   Config files
-   Logs
-   Knowledge base articles

In our example, this file is a resource:

data/project\_status.json

Resources help the AI agent understand the situation.

## Prompts

Prompts are reusable instruction templates.

Examples:

-   Summarize this project
-   Review this code
-   Generate a risk report
-   Explain this error
-   Create an action plan

Prompts guide the AI agent.

So the simple formula is:

Tools = actions  
Resources = context  
Prompts = instructions

When you understand this, MCP becomes much easier.

## The Security Problem Nobody Should Ignore

The more powerful AI agents become, the more dangerous bad tool design becomes.

This is very important.

If your AI agent can only answer questions, the risk is limited.

But if your AI agent can call tools, read files, access APIs, update systems, or run commands, the risk becomes much bigger.

A powerful AI agent needs strong boundaries.

When building an MCP server in Python, you should think about security from day one.

Here are some basic rules:

## 1\. Do not expose dangerous tools too early

Do not start with tools that delete files, update databases, send emails, or run shell commands.

Start with read-only tools.

## 2\. Validate every input

Never trust raw input.

If a tool expects a number, validate it.

If a tool expects a file path, restrict it.

If a tool expects a project name, check if it exists.

## 3\. Return structured output

Do not return messy text if JSON is better.

Structured output is easier to test, log, and debug.

## 4\. Log every tool call

For every tool call, you should know:

-   What tool was called
-   What input was used
-   What output was returned
-   When it happened
-   Whether it failed

## 5\. Use human approval for risky actions

Some actions should not be automatic.

Examples:

-   Sending email
-   Deleting records
-   Updating production data
-   Making payments
-   Changing access permissions

For risky actions, the AI agent should ask for approval.

## 6\. Keep secrets away from prompts

Never expose API keys, database passwords, tokens, or private credentials inside prompts.

Secrets should stay in environment variables or secure secret stores.

## 7\. Separate dev and production

Do not test a new AI agent directly on production systems.

Use separate environments.

Your MCP server should be safe by design.

A powerful agent without safety is not innovation.

It is a future incident report.

## Common Mistakes When Building MCP Servers

After building MCP-style tools, these are the mistakes I would avoid.

## Mistake 1: Giving the agent too many tools

More tools do not always mean a better agent.

Sometimes too many tools confuse the system.

Start small.

Add tools slowly.

## Mistake 2: Bad tool names

Tool names matter.

A tool called `process_data()` is vague.

A tool called `estimate_project_cost()` is clear.

Good tool names help the AI agent choose the right action.

## Mistake 3: Poor tool descriptions

The description should explain what the tool does and when to use it.

If the tool description is weak, the agent may call it incorrectly.

## Mistake 4: Letting the LLM do calculations

Do not let the LLM calculate important values.

Use Python.

Use tested functions.

Use deterministic logic.

## Mistake 5: Returning messy text

Return structured data whenever possible.

JSON is easier to debug than random paragraphs.

## Mistake 6: No logs

If you do not log tool calls, you will not know what happened.

And when the AI agent makes a mistake, debugging becomes painful.

## Mistake 7: Ignoring security

This is the biggest mistake.

If an AI agent can act, then tool permissions matter.

A badly designed tool can become a serious risk.
