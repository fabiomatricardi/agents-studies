The goal is not just “free access”. The actual design idea is to decouple Claude Code from a single model provider and make it backend-agnostic without modifying the Claude Code client.

## System Architecture Overview

At a high level, the system is a reverse proxy with request transformation and provider abstraction layers.

The pipeline looks like this:

Claude Code (CLI / VSCode)  
        ↓  
Free Claude Code Proxy (FastAPI server)  
        ↓  
Provider Router (NIM / OpenRouter / DeepSeek / Local LLMs)  
        ↓  
LLM Response Stream  
        ↓  
Claude-compatible SSE response back to client

The key design choice here is that Claude Code still believes it is talking to Anthropic. The proxy ensures full API compatibility by emulating Anthropic’s message format and streaming protocol.

## Request Flow and Transformation Layer

Every request from Claude Code follows a strict flow:

> Request ingestion (Anthropic format)
> 
> Pre-processing and optimization checks
> 
> Provider selection (routing logic)
> 
> Format conversion (if required)
> 
> Streaming execution
> 
> Post-processing (thinking tokens, tool calls)
> 
> Response streaming back to Claude Code

The important part is that the system does not blindly forward requests. It actively modifies or intercepts them.

Example flow:

POST /v1/messages  
{  
  "model": "claude-3-opus",  
  "messages": \[...\]  
}

The proxy transforms this into something like:

provider = NVIDIA\_NIM  
model = z-ai/glm4.

Then converts payload into provider-specific schema.

## Provider Abstraction Layer

The system supports multiple providers through a unified interface. Each provider implements a common abstraction:

-   OpenAI-compatible chat API OR
-   Anthropic Messages API OR
-   Local server (Ollama / LM Studio / llama.cpp)

Supported backends include:

-   NVIDIA NIM (free tier, rate-limited but stable)
-   OpenRouter (multi-model aggregation layer)
-   DeepSeek API (native Anthropic-compatible endpoint)
-   LM Studio (local OpenAI-style server)
-   llama.cpp server (lightweight local inference)
-   Ollama (native local model runtime)

Each provider is wrapped in a transport layer that normalizes:

-   Streaming format (SSE handling)
-   Token usage metadata
-   Tool call structures
-   Error formats (429, 500, timeout handling)

## Model Routing System

One of the core components is per-model routing. Instead of a single backend, different Claude “tiers” can map to different providers.Example configuration:

MODEL\="nvidia\_nim/z-ai/glm4.7"  
MODEL\_OPUS\="open\_router/anthropic/opus"  
MODEL\_SONNET\="deepseek/deepseek-chat"  
MODEL\_HAIKU\="ollama/llama3"

This allows a hybrid compute strategy:

-   Heavy reasoning → stronger cloud models
-   Medium tasks → balanced API models
-   Lightweight tasks → local inference

Routing logic is applied at request time, not compile time.

## Request Optimization and Cost Reduction

A major part of the system is request filtering. Not all Claude Code requests actually need LLM inference. The proxy identifies and intercepts trivial or redundant calls such as:

-   Token counting requests
-   UI suggestion generation
-   Filepath extraction
-   Prefix validation calls
-   Metadata-only probes

These are answered locally without hitting any LLM backend. This reduces:

-   API usage
-   Latency
-   Rate limit pressure

In practice, a significant portion of Claude Code traffic falls into these categories.

## Streaming and Protocol Compatibility

Claude Code relies heavily on streaming responses using Server-Sent Events (SSE). The proxy fully replicates this behavior. Key responsibilities:

-   Maintain chunked token streaming
-   Preserve event ordering
-   Translate provider-specific streaming formats into Anthropic format
-   Handle partial tool calls mid-stream
-   Ensure backpressure stability

For providers that do not natively support Anthropic streaming, the system emulates it by buffering and re-emitting chunks.

## Tool Call Parsing Layer

Some models do not output structured tool calls correctly. Instead, they return text-based instructions.The proxy includes a heuristic parser that detects patterns like:

<tool>  
name: file\_write  
args: {...}  
</tool>

or JSON-like fragments inside natural language. It converts these into proper Anthropic tool invocation objects before sending them back to Claude Code.

This makes weaker or non-Anthropic models usable inside Claude Code without breaking tool execution.

## Thinking Token Support

Claude-style reasoning outputs often include hidden reasoning or “thinking blocks”. The proxy supports multiple formats:

-   `<think>...</think>` tags
-   provider-specific reasoning fields
-   streaming reasoning chunks

These are normalized into Claude-compatible:

{  
  "type": "thinking",  
  "content": "..."  
}

This ensures reasoning visibility works consistently across all providers.

## Rate Limiting and Stability Layer

Since multiple providers have different constraints, the proxy includes a unified rate-limiting system. It uses:

-   Rolling window counters
-   Per-provider concurrency limits
-   Exponential backoff on 429 errors
-   Retry classification (retryable vs non-retryable errors)

Example config:

PROVIDER\_RATE\_LIMIT\=40  
PROVIDER\_RATE\_WINDOW\=60  
PROVIDER\_MAX\_CONCURRENCY\=5

This prevents cascading failures when one provider becomes unstable.

## Local vs Cloud Execution Strategy

The system is designed to support three execution modes:

### 1\. Fully cloud-based

Uses NVIDIA NIM or OpenRouter as primary backends.

### 2\. Hybrid mode

Cloud models for heavy tasks, local models for simple queries.

### 3\. Fully local mode

Uses Ollama / LM Studio / llama.cpp with no external API dependency.This makes the system flexible across:

-   low-cost setups
-   offline environments
-   high-performance cloud workflows

## Why This Architecture Works

The key insight is separation of concerns:Claude Code does not care about:

-   which model is used
-   where inference happens
-   how requests are routed

It only cares about:

-   Anthropic API compatibility
-   streaming correctness
-   tool execution integrity

The proxy ensures all three are preserved while making the backend interchangeable.

## Final View

This project is essentially a **runtime abstraction layer for LLM providers** tailored specifically for Claude Code.Instead of modifying the client or rewriting workflows, it:

-   intercepts API calls
-   rewrites them dynamically
-   routes them intelligently
-   preserves full protocol compatibility

In practical terms, it turns Claude Code into a multi-backend coding agent that can run on cloud APIs or local models without any client-side changes.
