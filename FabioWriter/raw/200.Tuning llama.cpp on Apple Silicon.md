## Most Parameters Don’t Matter. Here Are the Seven That Do.

A practical tuning guide from someone who got tired of cargo-culting flags from x86 tutorials.

> **TL;DR.** Seven flags do the work on Apple Silicon: `-ngl 99 -fa 1 -c <ctx> -b 2048 -ub 2048 --cache-type-k q8_0 --cache-type-v q8_0`. Add `--mlock --prio 2` if you have memory headroom. The single most important tuning step is not a llama.cpp flag at all, it is `sudo sysctl iogpu.wired_limit_mb=N`. Most other flags you find in tutorials were copied from x86 server documentation and have no effect on M-series chips.

In the [previous post](/@michael.hannecke/llama-cpp-vs-mlx-on-apple-mx-775ee59df0ee) I argued that the parameter gap between llama.cpp and MLX on Apple Silicon is mostly an artifact of where each project draws its abstraction boundary. Not all of llama.cpp’s flags exist to control real choices. A meaningful share of them manage hardware realities that Apple Silicon eliminated by design.

This post is the practical follow-up. If you actually run llama.cpp on a Mac, which flags matter, which can you safely ignore, and what does a sensible default configuration look like.

I will save you some scrolling. The answer is seven flags.

\-ngl 99 -fa 1 -c <ctx> -b 2048 -ub 2048 \\  
\--cache-type-k q8\_0 \--cache-type-v q8\_0

Add `--mlock --prio 2` if you have the memory headroom. That is the entire mandatory configuration on Apple Silicon. Everything else in the `--help` output is either situational, irrelevant on M-series chips, or actively counterproductive.

The rest of this post is the reasoning, plus the system-level hack that probably matters more than any flag, plus baseline configurations for the common scenarios.

## The seven flags

### `-ngl 99 or (--n-gpu-layers 99)`

This offloads all model layers to the Metal GPU. The number gets clamped to the actual layer count, so 99 is just shorthand for “everything.” On NVIDIA hardware this flag is the single most important performance lever. On Apple Silicon it is the same flag for a different reason. Without it, you fall back to the CPU path with NEON kernels, which is orders of magnitude slower.

The flag is named after NVIDIA conventions but applies to any GPU backend including Metal. There is no scenario on Apple Silicon where you want to leave layers on the CPU. Set it and forget it.

## `-fa on` (or `--flash-attn [on|off|auto]`)

Enables the Metal Flash Attention kernel. Apple Silicon has its own implementation of Flash Attention in llama.cpp, distinct from the original CUDA kernels by Tri Dao. The effect is real: faster prefill at long contexts, smaller memory footprint per token in attention, and it is a hard prerequisite for KV cache quantization.

If the model does not support Flash Attention, llama.cpp emits a warning and falls back. There is no downside to having the flag on by default. Apple’s own gpt-oss guide recommends it for M-series chips. I have not seen a configuration where leaving it off was the right choice.

## `-c <ctx>` (or `--ctx-size`)

Sets the maximum context window. Setting `-c 0` means "use the model's maximum," which sounds harmless until you load a Llama 3.1 model with its 128K theoretical context and your KV cache eats your memory before generation starts.

Pick a context size that matches your actual workload. For interactive chat, 8K or 16K is plenty. For document processing, 32K or 64K. The KV cache grows linearly with context, so the choice has direct memory consequences. On a 64 GB Mac running a 70B model, the difference between 16K and 32K context is whether you have room for `--mlock` and the rest of macOS.

## `-b 2048` and `-ub 2048 (or --batch-size, --ubatch-size)`

These are the logical batch size and the physical micro-batch size for prompt processing. The default is 512. On Apple Silicon, setting both to 2048 measurably accelerates the prefill phase by giving the Metal kernels more parallelism to work with during matrix multiplication.

Apple’s own gpt-oss guide explicitly recommends `-ub 2048 -b 2048`. The cost is a larger compute buffer during prefill, which can be a problem on memory-constrained machines but is generally fine if your model uses less than half of unified memory.

I noticed the difference most clearly on long prompts. A 4K-token prompt that took 8 seconds to process at default batch sizes drops to about 3 seconds with `-ub 2048`. For agent workflows where every interaction starts with a long system prompt, this adds up.

## `--cache-type-k q8_0 --cache-type-v q8_0`

Quantizes the KV cache to 8 bits. The default is `f16`. Allowed values include `f32`, `f16`, `bf16`, `q8_0`, `q4_0`, `q4_1`, `iq4_nl`, `q5_0`, and `q5_1`.

This halves the KV cache memory footprint. On a 14B model with 32K context, that is between 4 and 8 GB freed up. On a 64 GB Mac running a 70B model, it is the difference between fitting 16K and 32K context.

Two important constraints. First, this only works with `-fa 1`. Without Flash Attention, llama.cpp has to dequantize the cache for every attention computation, which can be slower than not quantizing at all. Second, value quantization is more sensitive than key quantization. q8\_0 for both is the safe choice with negligible quality loss (perplexity increase under 0.1). Going to q4\_0 for value is risky and worth measuring before you commit.

This is one of those flags where the upside is huge and the downside is essentially zero, as long as you remember the Flash Attention dependency.

## `--mlock`

Pins the model and KV cache in physical RAM and prevents macOS from paging them to disk. On Apple Silicon, the practical effect is that you do not get latency cliffs when memory pressure builds elsewhere on the system.

This is one of two flags where you actually need to think about your machine. If your model plus KV cache plus operating system overhead exceeds about 70 percent of unified memory, `--mlock` will cause the rest of your system to start thrashing. Browsers, IDEs, and background services will fight for the remaining memory. The 70 percent rule comes from Apple's own guidance and matches my experience.

If you are within budget, set it. If you are at the edge, leave it off and accept the swap risk, or buy more memory.

## `--prio 2` (or `--prio 3`)

Sets the scheduling priority of the llama.cpp process, with values 0 through 3. On macOS, this reduces the chance of the inference threads being interrupted by other workloads.

For dedicated inference workloads, set it to 2. Set it to 3 only if the machine is actually a dedicated inference server and you do not run anything else on it. The difference is small but measurable for long generations where interruption costs token timing.

That is the seven. Everything else is a footnote.

## The wired memory hack that probably matters more

`sudo sysctl iogpu.wired_limit_mb=N`

This is not a llama.cpp flag. It is a macOS kernel parameter, and it is the single most important tuning step on Apple Silicon for large model inference, and the one I see practitioners miss most often.

The Apple GPU driver allocates from a pool of wired (non-pageable) memory. The default limit is conservative. On a 64 GB Mac, the wired limit is typically around 48 GB out of the box. Anything beyond that for GPU allocations gets rejected. So even if you have a 70B model that physically fits in your 64 GB of unified memory, the GPU driver will not let you allocate it, and you fall back to CPU paths or fail outright.

The fix is to raise the limit explicitly:

\# 64 GB Mac, raise to 45 GB usable for GPU  
sudo sysctl iogpu.wired\_limit\_mb=46080  
  
\# 128 GB Mac, raise to 90 GB  
sudo sysctl iogpu.wired\_limit\_mb=92160  
  
\# 192 GB Mac, raise to 180 GB (Apple's own example)  
sudo sysctl iogpu.wired\_limit\_mb=184320

Apple’s general recommendation is no more than 70 percent of physical RAM. Above that, macOS itself starts struggling because it needs some memory for system processes that cannot be wired.

Note that this is not persistent. After a reboot, the limit resets. For persistent configuration, you can write a sysctl entry or use a LaunchDaemon. I have a script that runs it at login on my Mac Studio because I forget otherwise.

I cannot count the number of “why can’t I run a 70B model on my 64 GB Mac” questions that boil down to this. If you are about to file a GitHub issue about Metal allocation failures, check `sysctl iogpu.wired_limit_mb` first.

## Flags that have no effect on Apple Silicon

These are the ones you can drop from any config you find online. If you see them in a tutorial, the tutorial was probably written for x86 servers.

`--numa <strategy>`. NUMA optimizations for multi-socket x86 systems. Apple Silicon is not NUMA in that sense. The unified memory pool is one physical thing. The flag is at best a no-op, at worst it confuses the macOS scheduler.

`--main-gpu`, `--tensor-split`, `--split-mode`. Multi-GPU coordination. A Mac Studio has one GPU. Multi-Mac inference is not a llama.cpp feature, it is an MLX feature via `mx.distributed`. Drop these.

`--cpu-mask`, `--cpu-strict`, `--poll`. CPU affinity controls. macOS does not respect Linux-style CPU affinity in any meaningful way. The macOS scheduler manages performance and efficiency cores by its own logic, and the flag is roughly a placebo.

`--no-kv-offload`. Tells the engine to keep the KV cache on CPU rather than GPU. On Apple Silicon, where CPU and GPU share physical memory, this is conceptually unclear and operationally bad. Setting it forces the cache out of Metal-friendly buffer formats and slows attention down. Drop it.

CUDA and ROCm flags. Anything with `--cuda-` or matching ROCm conventions only exists in non-Metal builds. If you build llama.cpp with `-DGGML_METAL=ON`, those flags are not even there.

`-t N` with N greater than the number of performance cores. More threads than P-cores hurts. On an M2 Max (8 P-cores) or M3 Max (12 P-cores), setting `-t 16` puts work on efficiency cores that ends up blocking the performance cores. The default is usually correct. Apple's own benchmarks for gpt-oss use `-t 1 -fa 1`, because the GPU does the work and threading on the CPU side barely matters.

## Situationally useful flags

A few flags do have effect but are not always needed. Worth knowing they exist.

`-cb` and `-np N` for `llama-server`. Continuous batching and the number of parallel slots. If you are running a single-user setup, ignore them. If you are exposing the server to multiple agents or multiple users, set them explicitly. Each slot needs its own KV cache region, so memory scales linearly with `-np`.

`-md <model>` for speculative decoding. Use a smaller draft model whose proposals get verified by the main model in a single forward pass. Apple's own Recurrent Drafter research achieved 2.3x speedup on Metal. The output is mathematically identical to standard generation, so quality is not a concern. The cost is loading two models. On a 64 GB Mac with a 70B main model, this gets tight fast, because draft plus main plus KV cache together can blow through the 70 percent guideline.

`--no-mmap`. Disables memory-mapped loading. In most cases mmap is faster, but there is a known bug where loading hangs at around 75 percent on certain Apple Silicon setups, and `--no-mmap` is the documented workaround. Do not set it preemptively, only when you actually hit the hang.

`--rope-scaling`, `--rope-freq-base`, `--rope-freq-scale`, `--yarn-*`. Context extension via RoPE manipulation. Modern models (Llama 3.1+, Qwen 3+) handle context extension during training and do not need these flags. Older models can be extended at inference time, with quality cost. If you are running anything from 2024 or later, ignore them.

`--keep N`. Retains the first N tokens of the prompt during context shift. Useful for long-running agents with a fixed system prompt that should not get evicted. Set it explicitly for production agents, ignore it for chat.

## Sampling: hardware does not enter into it

llama.cpp’s sampling parameters are CPU post-processing on the logits. Apple Silicon is not relevant. Every sampler works identically on every backend. The choice is purely task-driven.

For most tasks, the standard set works:

\--temp 0.7 \--top-p 0.9 \--top-k 40 \--min-p 0.05 \--repeat-penalty 1.1

For reasoning models like gpt-oss, DeepSeek-R1, and QwQ, you want the opposite. Turn the samplers off:

\--temp 1.0 \--top-p 1.0 \--top-k 0 \--min-p 0

This is counterintuitive. The instinct is that filtering low-probability tokens improves output quality. For reasoning models, it does not, because they are trained on raw sampling distributions. Filtering interferes with their internal reasoning trajectory. I learned this the hard way when a chain-of-thought model produced visibly worse results with my standard sampler stack.

For creative writing and narrative tasks, llama.cpp’s sampler variety pays off:

\--dry-multiplier 0.8 \--dry-base 1.75 \--dry-allowed-length 2 \\  
\--xtc-probability 0.5 \--xtc-threshold 0.1 \\  
\--min-p 0.02

DRY plus XTC at default settings is the established combination for fiction and roleplay. XTC removes the most predictable tokens with a configurable probability, which breaks phrasing loops without sacrificing coherence. DRY catches sequence-level repetitions that simple repeat penalties miss.

For structured outputs, use `--grammar-file` or `--json-schema`. This is one of the genuine gaps in MLX, so if you need guaranteed JSON, llama.cpp is the answer.

## Baseline configurations

Pulling all of this together, here are the configs I actually use.

### Single-user CLI on 64 GB Mac, 30B model, long context

llama-cli \\  
  -m models/qwen3-30b-q4\_k\_m.gguf \\  
  -ngl 99 -fa 1 \\  
  -c 32768 -b 2048 -ub 2048 \\  
  \--cache-type-k q8\_0 \--cache-type-v q8\_0 \\  
  \--mlock \--prio 2 \\  
  \--temp 0.7 \--top-p 0.9 \--min-p 0.05

### Single-user CLI on 64 GB Mac, 70B model, moderate context

llama-cli \\  
  -m models/llama-3.3\-70b-q4\_k\_m.gguf \\  
  -ngl 99 -fa 1 \\  
  -c 16384 -b 2048 -ub 2048 \\  
  \--cache-type-k q8\_0 \--cache-type-v q8\_0 \\  
  \--mlock \--prio 2

Run `sudo sysctl iogpu.wired_limit_mb=46080` first.

### Multi-client server

llama-server \\  
  -m models/qwen3-14b-q4\_k\_m.gguf \\  
  -ngl 99 -fa 1 \\  
  -c 16384 -b 2048 -ub 2048 \\  
  \--cache-type-k q8\_0 \--cache-type-v q8\_0 \\  
  -np 4 -cb \\  
  \--prio 2 \\  
  \--host 0.0.0.0 \--port 8080 \\  
  \--jinja

`--jinja` activates Jinja chat templates, which matter for modern models, especially with tool calling.

### Reasoning model (gpt-oss)

llama-server -hf ggml-org/gpt-oss-20b-GGUF \\  
  \--ctx-size 0 \--jinja \\  
  -ngl 99 -fa 1 \\  
  -ub 2048 -b 2048 \\  
  \--temp 1.0 \--top-p 1.0 \--top-k 0 \--min-p 0

### Speculative decoding

llama-cli \\  
  -m models/qwen3-14b-q4\_k\_m.gguf \\  
  \--model-draft models/qwen3-0.6b-q8\_0.gguf \\  
  -ngl 99 -ngld 99 -fa 1 \\  
  -c 16384 -b 2048 -ub 2048 \\  
  \--cache-type-k q8\_0 \--cache-type-v q8\_0 \\  
  \--draft-max 16 \--draft-min 5 \\  
  \--mlock \--prio 2

Note `-ngld 99` to offload the draft model layers as well.

## Why this matters

The gap between “what llama.cpp exposes” and “what actually does something on Apple Silicon” is a useful filter for evaluating any tutorial you read. A guide that tells you to set `--numa distribute` on a Mac is a guide that was copied from an x86 deployment doc without thinking. A guide that does not mention the wired memory limit on a 64 GB+ Mac is a guide written by someone who has not actually run a 70B model.

The seven flags above plus the sysctl hack get you 95 percent of the achievable performance. The remaining 5 percent is workload-specific tuning that you discover by running benchmarks on your actual prompts, not by adding flags from someone else’s config.

If you take one thing from this, take the wired memory limit. The number of times that has been the actual answer to “why can’t I run this model” is too high to keep being a footnote.

Default to fewer flags. Add complexity only when something is actually slow.
