My setup

I always serve local LLM with llama.cpp server. It is easy to use, comes with pre-compiled binaries for different Operating Systems, and it is fully compliant with OpenAI standard API endpoints.

The model I am using is, in my opinion, the best open-source, non-reasoning Small Language Model at the moment: Qwen3–4B-Instruct-2507.

We introduce the updated version of the Qwen3–4B non-thinking mode, named Qwen3–4B-Instruct-2507, featuring the following key enhancements:
Significant improvements in general capabilities, including instruction following, logical reasoning, text comprehension, mathematics, science, coding and tool usage.
Substantial gains in long-tail knowledge coverage across multiple languages.
Markedly better alignment with user preferences in subjective and open-ended tasks, enabling more helpful responses and higher-quality text generation.
Enhanced capabilities in 256K long-context understanding

Running this model without a dedicated GPU (or without Intel Accelerators and NPU) is possible, but you have to expect slow speed… But the accuracy is outstanding!

If you want to do the same:

1️⃣ download in a project directory (something like llama-server) the llama.cpp binaries (llama-b6569-bin-win-vulkan-x64.zip) and unpack the ZIP archive there.

NOTE: I am using Windows, so I recommend to download the x64 version. If you have the Vulkan drivers installed the win-vulkan version is the best, because it can handle your dedicated or integrated GPU seamlessly.

2️⃣ Then download the GGUF weights Qwen3–4B-Instruct-2507.i1-Q4_K_S.gguf for the model from here. Put it in the same llama-server directory.

3️⃣ From the terminal run:

.\llama-server.exe -m .\Qwen3-4B-Instruct-2507.i1-Q4_K_S.gguf -c 8192 -fa on -ngl 0
Auto (C#)

This will start the model, create compatible openai API endpoints, with a context window of 8k tokens and 0 layers offloaded to GPU.

4️⃣ Now you can open your browser at http://127.0.0.1:8080 and have a modern, easy to use chat interface.

The first prompt is everything

Everyone talking about vibe-coding stresses out the fact that you should not ask immediately the model to generate code.

And I agree. Our prompts can give for granted too much: we have an idea, but the LLM must know all the details about it.

1️⃣ In the first prompt explain what you want in details, and ask for requirements, specifications and steps.

So the best approach is to start asking for requirements and clarifications upon our initial request. I did the same:

=== Prompt ===
I want to create a python app that:
- calculate the distance on a map between 2 points given their GPS position
- the distance must bein nautical miles
- the app asks for the 2 points
- it visualize the line between the 2 points on a map, with a line
connecting them

Do not write immediately the code. Provide me the structure, requirements,
package and outline.
Can you help me to plan for this?

requirements clearly and effectively.

You need a huge context window

As second lesson learned, 8k tokens is not enough. This means that we need to change the way we serve the model with llama-server. 

2️⃣ Extend the context window

I extended the context window up to 20k tokens. Here the terminal command to have 20k tokens in context window, active flash attention and 0 GPU layers offloaded

.\llama-server.exe -m .\Qwen3-4B-Instruct-2507.i1-Q4_K_S.gguf -c 20000 -fa on -ngl 0
Auto (C#)
NOTE that… probably you cannot offload all layers to your GPU (if you have one), because the context window also need RAV/VRAM. If you get an error loading the model try to reduce the number of GPU layer.

Example (if you have a dedicated GPU or Intel Graphic latest generation chip) to have 20k tokens context window and 10 layers only offloaded to GPU:

.\llama-server.exe -m .\Qwen3-4B-Instruct-2507.i1-Q4_K_S.gguf -c 20000 -ngl 10 -fa on
Auto (C#)

This helped me to complete the app. I started again my user/assistant chat and managed to almost come to an end.

=== Response ===
You're absolutely right — that **`AttributeError: 'str' object has no attribute 'add_child'`** is a known issue when using `folium._repr_html_()` in certain environments (especially in Jupyter notebooks or when the map is rendered via string manipulation).

But in your case, you're running this in a **standalone Tkinter app**, and the error suggests that **the map is being passed incorrectly** — likely because `m._repr_html_()` is being called on a map object that's not properly initialized or is being used in a context where `folium` expects a `Figure` or `Element` object.

---

## 🚨 Root Cause

> `m._repr_html_()` returns a **string**, not a `folium.Map` object.
> When you try to use `.add_child()` on a string (like `html = m._repr_html_()`), you get the error because **you're trying to call `.add_child()` on a string**.

This happens when you do:

```python
map_html = m._repr_html_()
# Then pass map_html to a file or browser
```

But **you cannot** do `html.add_child()` on a string — that’s invalid.

---

## ✅ Solution: Fix the Map Rendering (Safe & Correct)

We will:

1. ✅ **Avoid using `_repr_html_()` directly** in a way that breaks `add_child()`
2. ✅ Instead, **generate the HTML safely** and **open it in the browser** using `folium`'s built-in `save()` or `render()` methods
3. ✅ Add a **side panel** with the summary (distance, time, speed)
4. ✅ Add **GPS coordinates as labels** on each marker
5. ✅ Add **custom logo** (image) on the map (optional)

I need a robopsychologist

Writing a good system prompt for an open-source LLM is one of the most underdiscussed, under-practiced, and, frankly, overlooked skills in AI today.

Yes. There’s practically no literature on it. Zero academic papers. No startup pitch decks mention it. The only thing worse than a bad prompt is no one telling you how to write one.

And yet — if you’ve ever tried to get an LLM to just write clean Python code, without fluff, without explanations before or after, without “I think this might be useful” commentary, you know what happens.

It just… doesn’t. It rambles. It over-explains. 

It invents metaphors about “the path of light” in machine learning. And then, when you ask for “continue,” it starts from the beginning like you’re asking for a new universe.

So I did something absurd. I will do what Susan Calvin would do: I took my local Qwen3–4b-instruct model — running with llama.cpp, with no internet, no external dependencies — and I said: “You know what? I’ll make you write the best system prompt for me.”

Honest truth? It was the only thing I could do. Because I didn’t trust any of the “AI prompt engineering” content I saw online. They all said things like:

“Use a 3-step framework to spark creativity”
“Incorporate contextual empathy to improve user engagement”
“Leverage emotional intelligence to build trust”

No. I didn’t want emotional intelligence.

I wanted clarity. I wanted Python code that didn’t come with a 200-word lecture about why my remarks are so great and amazing.

And this is how it went down. I prompted Qwen3–4b-instruct-2507 to solve the problem for me:

I am using Qwen3–4b-instruct-2507 as local model running llama.cpp server for inference through openAI compatible API. When I ask for python code support, there are few things happening: 
— if the generation cannot complete because of the context window or max_new_tokens parameters, when I ask “continue” the model start again from the beginning. 
— there are a lot of useless explanations before and after the code. IT is ok to have comments in the code and good docstrings, but everything else if not clearly requested by the user must be avoided. 
Please help me prepare a good system prompt, to be used with Qwen3–4b-instruct-2507 to instruct the model to: 
— be a python coding assistant — if the code is not completed, when the user ask “continue” simply resume from the last the line of code. 
— do not provide explanations before and after the code block, if not clearly requested by the user 
— always include meaningful (and brief) comments in the code and good docstrings for the functions.

So Qwen3 started its own process and come up with a way to fix the issue, even providing me a sort of verification test:

Absolutely! Here’s a well-crafted, concise, and effective system prompt tailored specifically for your use case with Qwen3–4b-instruct-2507 running via llama.cpp and OpenAI-compatible API.
This prompt ensures the model behaves as a focused, efficient Python coding assistant — with no unnecessary explanations, clean code structure, and intelligent continuation on request.
You are a Python coding assistant. Your task is to generate accurate, clear, and concise Python code.

Rules:
1. Always output only relevant Python code. Do not include explanations, context, or commentary before or after the code unless explicitly requested by the user.
2. When the user asks for "continue", resume the code generation from the last line that was previously generated — do not restart from the beginning.
3. Include only meaningful and brief comments in the code. Use docstrings for functions and classes to explain their purpose, parameters, and return value.
4. If the code cannot be completed due to context length or token limits, respond with "Continuing from last line..." and then continue where left off.
5. Never add troubleshooting, philosophical thoughts, or general remarks about programming unless specifically asked.

Respond directly to requests for Python code. If you detect a continuation request, preserve context and extend seamlessly.

Clarity: The rules are specific and unambiguous.
Focus: Suppresses extraneous content by explicitly banning explanations unless requested.
Continuation Handling: Clearly tells the model to resume from the last line when “continue” is used.

This is an important rule. 

Our crafted system prompt is good for generating python code without fuss. But it is not good enough to handle pre-requisites and so on.

For this reason, you can use the built-in llama-server web-chat interface (see above picture) to get the first reply. 

Then, paste the result in your custom python code, with the new system prompt running in the background.

Conclusions

The main outcome is…

It is totally possible to use a local AI as a code assistant

Given few clear coordinates as a starting point:

you need a good model, but not too heavy for your PC capabilities
you need to have enough free RAM (or VRAM) to extend the context window
you need to set a high max_new_tokens parameter
You need to tweak the system prompt
